# i18n Implementation Plan for Zentrix Landing Page

## 1. Overview

This plan outlines the implementation of internationalization (i18n) for the Zentrix landing page project, migrated from Astro to Next.js 16.2.3 with App Router. The goal is to support both English (en) and Vietnamese (vi) locales.

## 2. Project Context

### Current State
- **Framework**: Next.js 16.2.3 with App Router
- **React**: 19.2.4
- **Styling**: Tailwind CSS v4 (CSS-based config with `@import "tailwindcss"` and `@theme` directive)
- **TypeScript**: 5 in strict mode
- **Path Alias**: `@/*` maps to project root
- **Fonts**: Geist and Geist_Mono via `next/font/google`
- **ESLint**: 9 with flat config format

### Previous Implementation (Astro)
The original Astro project had i18n files that need to be migrated:
- `src/i18n/en.json` - English translations
- `src/i18n/vi.json` - Vietnamese translations
- `src/i18n/utils.ts` - i18n utility functions

### Landing Page Content
The current page.tsx contains:
- "Create Next App" title
- Template selection link
- Learning center link
- "Deploy Now" and "Documentation" buttons
- Next.js logo, Vercel logomark

## 3. Recommended Solution: next-intl

### Why next-intl?

| Criteria | next-intl | react-intl | react-i18next |
|----------|-----------|------------|---------------|
| App Router Support | First-class | Partial | Partial |
| Bundle Size | ~2KB gzipped | ~12KB | ~8KB |
| Type Safety | Excellent | Good | Good |
| ICU Message Format | Yes | Yes | Yes |
| Maintenance | Active | Active | Active |
| Next.js 16.x Support | Yes | Unknown | Yes |
| Learning Curve | Low | Medium | Medium |

**next-intl is the recommended choice** because:
1. App Router-first design
2. Lightweight (~2KB)
3. Excellent TypeScript support with automatic type generation for translation keys
4. Native support for ICU message syntax
5. Seamless integration with Next.js middleware
6. Supports locale-based routing out of the box
7. Active maintenance and large community

## 4. Architecture

### 4.1 Routing Strategy

Two options for locale handling:

**Option A: Locale in Path (e.g., `/en/`, `/vi/`)**
- Pros: SEO-friendly, clear URL structure
- Cons: Requires restructuring app directory to `app/[locale]/`

**Option B: Locale via Cookie/Header (e.g., `/`)**
- Pros: No URL restructuring needed
- Cons: Less SEO-friendly, requires cookie handling

**Recommendation**: Option A (Locale in Path) for a public landing page where SEO matters.

## 5. Implementation Steps

### Phase 1: Setup and Configuration

#### Task 1.1: Install next-intl
```bash
npm install next-intl
```

#### Task 1.2: Configure next.config.ts
```typescript
// next.config.ts
import type { NextConfig } from "next";
import createNextIntlPlugin from "next-intl/plugin";

const nextConfig: NextConfig = {};

const withNextIntl = createNextIntlPlugin();
export default withNextIntl(nextConfig);
```

#### Task 1.3: Create routing configuration
```typescript
// src/i18n/routing.ts
import { defineRouting } from "next-intl/routing";

export const routing = defineRouting({
  locales: ["en", "vi"],
  defaultLocale: "en",
  localePrefix: "always" // or 'as-needed' if you want root / for default
});
```

#### Task 1.4: Create request configuration
```typescript
// src/i18n/request.ts
import { getRequestConfig } from "next-intl/server";
import { routing } from "./routing";

export default getRequestConfig(async ({ requestLocale }) => {
  let locale = await requestLocale;

  if (!locale || !routing.locales.includes(locale as "en" | "vi")) {
    locale = routing.defaultLocale;
  }

  return {
    locale,
    messages: (await import(`../../messages/${locale}.json`)).default
  };
});
```

#### Task 1.5: Create middleware
```typescript
// middleware.ts
import createMiddleware from "next-intl/middleware";
import { routing } from "./src/i18n/routing";

export default createMiddleware(routing);

export const config = {
  matcher: [
    // Match all pathnames except for
    // - … if they start with `/api`, `/trpc`, `/_next` or `/_vercel`
    // - … the ones containing a dot (e.g. `favicon.ico`)
    "/((?!api|trpc|_next|_vercel|.*\\..*).*)"
  ]
};
```

#### Task 2.3: Create locale layout
```typescript
// app/[locale]/layout.tsx
import { NextIntlClientProvider } from "next-intl";
import { getMessages, getTranslations } from "next-intl/server";
import { notFound } from "next/navigation";
import { routing } from "@/i18n/routing";

type Props = {
  children: React.ReactNode;
  params: Promise<{ locale: string }>;
};

export async function generateStaticParams() {
  return routing.locales.map((locale) => ({ locale }));
}

export async function generateMetadata({ params }: Props) {
  const { locale } = await params;
  const t = await getTranslations({ locale, namespace: "Common" });

  return {
    title: t("appName"),
    description: t("loading") // Update with actual description
  };
}

export default async function LocaleLayout({ children, params }: Props) {
  const { locale } = await params;

  // Validate locale
  if (!routing.locales.includes(locale as "en" | "vi")) {
    notFound();
  }

  const messages = await getMessages();

  return (
    <html lang={locale}>
      <body>
        <NextIntlClientProvider messages={messages}>
          {children}
        </NextIntlClientProvider>
      </body>
    </html>
  );
}
```
